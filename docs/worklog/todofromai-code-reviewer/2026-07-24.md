## What I checked
- **No double-send:** the atomic `UPDATE ... SET status='sent' WHERE id=? AND status='pending'
  RETURNING` claim is the correct pattern — exactly one caller wins the row under concurrency; the
  race test proves it. Revert-to-pending on delivery failure gives at-least-once without dropping.
  Good.
- **Auth:** constant-time compare via `timingSafeEqual` with a length guard; fails closed (503 when
  unset, 401 on mismatch); rate-limit disabled only for this route. Secret never inlined — env +
  Cloud Secret Manager ref. Correct and safe to ship ahead of the secret (endpoint is a 503 no-op
  until set).
- **Digest idempotency:** `lastSentOn` guard is necessary given the frequent tick; stamping even on
  an empty agenda avoids recomputing the (potentially AI-written) digest every tick. Reasonable.
- **Removal of in-process crons:** reminder firing + poll/digest/cleanup/billing removed from the
  scheduler; `sync` retained (user-triggered, keeps instance awake). No dangling references to the
  removed queues/handlers (grep clean). `enqueue` no-op keeps call sites intact; unused `QUEUES`
  import dropped. Build + typecheck clean.

## Observations (non-blocking)
- Inbound polling relaxes from every 2 min to the tick interval (~10 min). Acceptable and
  consistent with the cost decision; flag if inbound-sync latency ever matters.
- The digest pass scans all users each tick (`findMany`). Fine at current scale; revisit if the
  user table grows large (index/paginate or a due-users query).
