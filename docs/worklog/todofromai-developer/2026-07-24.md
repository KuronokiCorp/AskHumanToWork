## What I built
- `packages/api/src/reminders-runner.ts` — `deliverDueReminders(ctx)`: selects `status='pending'
  AND fireAt<=now` (uses the existing `reminders_fire_idx`), delivers each, and returns
  `{processed, failed, deferred}`. Atomic **pending→sent claim** before send so overlapping ticks
  or a still-awake pg-boss worker can never double-send; **revert to pending on delivery failure**
  so nothing is silently dropped (at-least-once). Kept the original delivery behaviour verbatim:
  quiet-hours defer, overdue wording, HMAC one-click action links, overdue-nudge escalation,
  done/cancelled short-circuit.
- `packages/api/src/cron-runner.ts` — `runCronTick(ctx)`: reminders + digest + inbound poll +
  session cleanup + Stripe usage, each independently try/caught. Digest gets a **once-per-day
  guard** (`lastSentOn` written into `notificationPrefs.digest`) because the tick runs every
  ~10 min and `isLocalHour` is true for the whole target hour.
- `packages/api/src/routes/cron-routes.ts` — `POST /api/internal/cron/tick`. Constant-time secret
  compare (`timingSafeEqual`), accepts `Authorization: Bearer` or `X-Cron-Key`. Fails closed:
  503 when `CRON_SECRET` unset, 401 on bad/missing key. Rate-limit disabled like `/api/health`.
- `packages/api/src/digest.ts` — added `lastSentOn` to `DigestPrefs` + a `localDateOf()` helper.
- `packages/api/src/worker.ts` — removed reminder firing and the poll/digest/cleanup/billing
  schedules from the in-process scheduler; only the on-demand `sync` outbox drain remains.
- `packages/core/src/reminder-service.ts` — `enqueue()` is now a no-op (table is the firing source
  of truth); dropped the now-unused `QUEUES` import. All call sites unchanged.
- Config/docs: `CRON_SECRET` in `env.ts`, `.env.example`, `apphosting.yaml` (secret ref, commented
  until the deploy step creates it); `docs/runbooks/cloud-scheduler.md` for the CEO-approved deploy.
